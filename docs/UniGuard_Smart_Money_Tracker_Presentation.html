Slide 2 — Introduction and background

Personal financial management (PFM) has moved from paper ledgers to spreadsheets and mobile apps. Users expect fast data entry, automatic categorization, and actionable feedback. At the same time, students and professionals in emerging markets—including Uganda (UGX)—need tools that work with everyday receipts, informal spending descriptions, and varying income patterns.

UniGuard Smart Money Tracker is a full-stack web application that addresses this need. It combines:

  • A modern React single-page application (Vite, TypeScript, Tailwind, Shadcn/Radix) for responsive UX.
  • Supabase (PostgreSQL with Row Level Security) for authentication and durable user data.
  • A Python Flask service exposing REST endpoints for text categorization, anomaly scoring, and receipt scanning.
  • Offline-trained scikit-learn pipelines (TF–IDF + Random Forest, etc.) packaged as joblib artifacts.
  • Optional DocTR-based OCR (with Tesseract fallback) so receipt images can become structured inputs without paid cloud OCR APIs.
  • Hybrid AI: backend ML for categorization and anomaly detection; frontend logic for forecasts, goal simulations, and companion-style insights—reducing round-trips for interactive features.

The project positions itself as an intelligent tracker: it helps users see where money goes, whether spending is unusual for them, and how goals might play out—rather than acting as a payment instrument.

Slide 3 — Problem statement

Despite the availability of generic budgeting apps, several gaps remain for a capstone-style, extensible system:

  1. Manual categorization burden: Users type free-text descriptions (“50k fuel”, “coffee 450”). Without ML, every transaction must be manually labeled for meaningful reports.
  2. One-size-fits-all anomaly rules: Fixed thresholds (e.g. “flag > 100,000”) fail across users and categories. An amount normal for one person may be extreme for another.
  3. Receipt friction: Paper and photo receipts contain the ground truth for spending, but transcribing them is tedious and error-prone.
  4. Fragmented insight: Budgets, goals, and transactions are most useful when connected—yet many tools separate “entry” from “planning” and “visualization.”
  5. Trust and explainability: Users need confidence scores, alternatives, and clear anomaly rationale—not opaque black boxes only.

The problem, therefore, is to design and implement an integrated PFM system that reduces manual work through ML and OCR, surfaces user-relative anomalies, and presents finances through a cohesive dashboard—including optional gamification to sustain engagement.

Slide 4 — Objectives

  1. Product naming and positioning: Present the system as UniGuard Smart Money Tracker (or chosen variant), reflecting tracking and AI assistance rather than a payment wallet.
  2. Transaction intelligence: Deploy a trained text categorization model (19 categories) with confidence and top alternatives; route low-confidence predictions to an “Other/Unknown” bucket.
  3. Anomaly awareness: Implement user-relative anomaly detection using features such as amount_ratio (amount ÷ user’s median for that category), combined with transaction type, category, and payment mode.
  4. Receipt digitization: Integrate OCR (DocTR primary, Tesseract fallback), regex-based field extraction, and per-line categorization where applicable.
  5. User experience: Deliver Financial Nexus (3D visualization), Budget Ports / Flux Pods, goals with probabilistic projections, reports, AI companion patterns, achievements, and optional email notifications for anomalies.
  6. Engineering rigor: Document architecture, API contracts, data flow, security (RLS), and limitations for future work.

Slide 6 — Literature review (concise)

Personal finance and PFM systems. Academic and industry literature describes PFM as combining data aggregation, categorization, budgeting, and visualization. Dashboards and trend charts improve awareness; automation reduces dropout.

Text classification for transactions. Short, noisy user phrases resemble micro-text classification. Classical pipelines—TF–IDF with Random Forest or linear models—remain strong baselines when labeled data are limited and interpretability matters. Confidence thresholds mitigate misclassification.

Anomaly detection in spending. Isolation Forest and tree ensembles appear in fraud and outlier detection. User-relative normalization (e.g., ratios to personal or category baselines) addresses scale differences across income levels—important in UGX and other currencies.

OCR for receipts. Deep learning detectors/recognizers (e.g., DocTR-style architectures) improve robustness on camera images; rule-based or regex post-processing extracts amounts, dates, and merchants from noisy text.

Gamification. Points, badges, and challenges (achievements, weekly challenges) can increase engagement; design should avoid “dark patterns” and align with financial well-being goals.

Slide 7 — Research methodology (adopted and why)

Approach: Design science / engineering research—build an artifact (the system), justify technology choices with literature and constraints, and evaluate through functional demonstration plus quantitative metrics where available (e.g., categorization accuracy from validation splits).

Why hybrid AI: Backend models handle tasks needing trained weights (categorization, anomaly). Frontend simulations and heuristics support rapid interaction (goal Monte Carlo–style projections, companion responses from precomputed insights) without blocking on network for every keystroke.

Why user-relative anomaly features: Global amount cutoffs are unfair across users. amount_ratio ties each transaction to the user’s own category history, improving fairness and portability across currencies.

Why DocTR + fallback: On-premise OCR avoids mandatory third-party API keys and recurring costs; Tesseract provides resilience when the DocTR service is down.

Data ethics: Training uses synthetic or de-identified patterns; production data stay user-scoped under RLS. Anomaly email alerts are optional and user-configured.

Appendix — Technical detail (for answers or handout)

Machine learning (summary)

  • Categorizer: TF–IDF + Random Forest; 19 categories; preprocessing (lowercase, stopwords, noise); low confidence → Other/Unknown.
  • Anomaly: Random Forest (with alternatives such as Decision Tree / Isolation Forest in pipeline work); features include transaction_type, category, amount_ratio, payment_mode; labels often derived from per-user tails of amount_ratio.
  • Artifacts: e.g. transaction_rf_pipeline.pkl, anomaly_rf_pipeline.pkl under ml_pipeline/ or backend training folders.

Core UX modules (from project report)

  • Financial Nexus: Three.js 3D scene; quick stats; wellness score; predictions feed; anomaly summary; time machine; notifications.
  • Transaction center: natural-language quick entry; AI recategorize; search/filter; receipt upload.
  • Budget Ports / Flux Pods: velocity, health states, AI forecasts and suggestions.
  • Goals: deadlines, contributions, Monte Carlo–style success visualization.
  • AI companion: intents mapped to dashboard insights; optional voice.
  • Reports, savings vault, family finance, investments, achievements.

Unique differentiators (talking points)

  1. Hybrid backend ML + frontend logic for responsiveness.
  2. 3D Financial Nexus as a distinctive visualization.
  3. Flux Pods with velocity and AI-assisted allocation narrative.
  4. On-premise OCR path without mandatory cloud API keys.
  5. Scale-invariant anomaly features for diverse users and currencies.
